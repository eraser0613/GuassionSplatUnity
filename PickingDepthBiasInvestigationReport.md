# 3DGS 测距点偏前问题调研报告

日期：2026-04-28

## 1. 背景

在当前 Unity Gaussian Splatting 测距工具中，用户点击 3DGS 画面后，系统会返回一个世界坐标作为测距点，并在场景中显示：

- 点击射线；
- 最终测距点；
- 候选高斯；
- 候选高斯的 alpha、contribution、accumulated alpha、depth 等诊断值。

经过前序调试，已经修复了一个关键坐标问题：picker 的屏幕坐标计算没有匹配渲染 shader 中的 backbuffer Y 翻转，导致候选高斯系统性偏低。修复后，候选位置与真实画面对齐明显改善。

本次调研关注新的现象：

> 即使坐标对齐后，有时最终测距点仍然比用户期望的真实点更靠前；即使将 `surfaceAlpha` 调到 1.00，也不能完全选到更深、更准确的位置。

## 2. 当前测距算法概述

当前稳定版算法是 **渲染一致的屏幕像素贡献拾取**，不是严格的 3D 几何射线求交。

核心流程：

1. 用户点击屏幕像素。
2. Picker 使用当前相机、当前 renderer 的 GPU view data 和 sort order。
3. Compute shader 按渲染顺序遍历 splat。
4. 对每个 splat 判断其屏幕投影椭圆是否覆盖点击像素。
5. 若覆盖，则计算该 splat 在该像素上的 alpha。
6. 根据透明混合公式计算该 splat 的可见贡献：

   ```hlsl
   contribution = (1.0 - accumA) * alpha;
   ```

7. 累积 alpha，并在达到 `surfaceAlpha` 前计算可见深度。
8. 使用候选 splat 的 `usedContribution` 对线性深度加权：

   ```hlsl
   weightedLinearDepthSum += linearDepth * usedContribution;
   contributionSum += usedContribution;
   finalLinearDepth = weightedLinearDepthSum / contributionSum;
   ```

9. 将该深度投回点击射线，得到最终测距点。

因此，最终测距点是：

> 点击像素处的透明混合可见表面深度。

而不是：

> 鼠标射线与某个真实 mesh 表面或 3D Gaussian 椭球的几何交点。

## 3. 观察到的问题

用户反馈和测试现象：

1. 有些区域测距比较准确；
2. 有些区域最终点明显偏前；
3. 将 `surfaceAlpha` 拉高到 1.00 后仍不能完全解决；
4. 相机离目标更近时，点击结果会更准确；
5. 偏差主要表现为深度方向上的“前移”，而不是屏幕 X/Y 坐标错位。

其中第 4 点非常关键：

> 离得近时更准确，说明问题不是单纯坐标翻转或像素映射错误，而更可能是透明高斯层叠造成的深度估计偏差。

## 4. 根因分析

### 4.1 前层透明高斯会稀释后层贡献

3DGS 渲染是大量半透明高斯按深度排序混合得到的。

对于点击像素，从前往后遍历 splat 时，前面的高斯会先累积 alpha：

```hlsl
accumA += contribution;
```

后面的 splat 贡献会被剩余透明度衰减：

```hlsl
contribution = (1.0 - accumA) * alpha;
```

这意味着：

- 前面较浅的高斯即使单个 alpha 不高，也会先占据一部分累计 alpha；
- 后面真正接近用户期望表面的高斯会被 `(1.0 - accumA)` 压低贡献；
- 最终加权深度容易被前层透明高斯拉浅。

这就是“偏前”的主要原因。

### 4.2 `surfaceAlpha` 调到 1.00 也不一定能解决

即使 `surfaceAlpha` 调高，算法仍然是一个累计透明混合过程。

当前的加权深度不是简单选择最深或最接近视觉中心的 splat，而是根据 `usedContribution` 做平均。

在透明混合中，前层高斯的权重天然更高；后层高斯虽然也参与，但权重会被前层 alpha 稀释。

因此，增大 `surfaceAlpha` 只能让更多后层贡献进入计算，但不能完全消除前层贡献对深度的拉浅作用。

### 4.3 近距离点击更准确的原因

用户观察到：

> 如果离得很近去点，则较为准确。

原因可能包括：

1. 近距离时，一个屏幕像素对应的世界空间范围更小；
2. 点击像素覆盖的高斯数量减少；
3. 前后层混合减少；
4. 目标区域的主导 splat 更集中；
5. 深度加权平均受到前层透明高斯干扰更小。

因此近距离点击时，当前屏幕贡献算法更容易接近用户期望的真实表面。

## 5. 当前结论

本次调研确认：

> 当前测距点有时偏前，主要不是坐标变换错误，而是 3DGS 半透明高斯层叠导致的深度估计偏差。

更具体地说：

- 前面的透明高斯会先累积 alpha；
- 后面的真实目标区域贡献被剩余透明度压低；
- 当前算法使用贡献加权深度，因此最终点容易被前层高斯拉到更靠前的位置；
- 离近后更准确，进一步支持这一判断。

## 6. 对当前调试界面的解释

当前调试显示中：

- 彩球/标签表示候选深度投到点击射线上的位置；
- 洋红高亮表示对应 Gaussian center；
- `alpha` 表示该 splat 在点击像素处的不透明度；
- `screenContrib` 表示该 splat 在当前透明混合顺序下的可见贡献；
- `used` 表示该 splat 对当前 surfaceAlpha 深度计算实际使用的贡献；
- `accum` 表示遍历到该 splat 后的累计 alpha。

如果看到前层候选 `used` 较大，而更深候选 `screenContrib` 或 `used` 很小，则说明后层目标正在被前层透明高斯稀释。

## 7. 后续优化方向

### 7.1 提供 Depth Bias 参数

可以在前端增加一个 `Depth Bias` 滑条，将最终点沿点击射线前后微调：

```text
finalDepth += depthBias
```

优点：

- 实现简单；
- 稳定；
- 可快速补偿整体偏前问题。

缺点：

- 是经验补偿，不是真正几何修正；
- 不同区域可能需要不同 bias。

### 7.2 提供候选选择模式

可以增加不同测距策略，例如：

1. 最大贡献候选；
2. surfaceAlpha 加权深度；
3. 更深层候选优先；
4. 累积 alpha 达到某阈值后的第一个强贡献候选；
5. 忽略前层弱 alpha 后再计算深度。

这样用户可以根据场景选择更合适的模式。

### 7.3 二阶段真实射线校正

理想方案是：

1. 先使用当前屏幕贡献 picker 找到少量候选；
2. 再只对这些候选执行轻量的 3D 几何分析；
3. 根据点击射线与候选 Gaussian 的几何关系重排或校正深度。

注意：之前尝试过全量 GPU 3D 椭球射线求交，但在打包环境中容易造成卡死或闪退。后续如果要做，应避免全量扫描几百万 splat，而应只处理已经筛出来的小候选集。

### 7.4 近距离测距作为当前工作流建议

在当前算法下，如果需要更准确的点位，建议：

- 先通过 WASD / 右键旋转靠近目标；
- 再点击测距；
- 必要时调节 `Surface Alpha`；
- 使用调试候选列表观察前层透明高斯是否在稀释目标。

## 8. 总结

当前偏前问题的核心原因是：

> 3DGS 的半透明前层高斯对点击像素有可见贡献，并在透明混合中优先累积 alpha，从而稀释后层目标贡献，导致加权深度偏向相机一侧。

这解释了为什么：

- 某些区域很准；
- 某些区域偏前；
- `surfaceAlpha = 1.00` 也不总能解决；
- 相机靠近后准确性提高。

当前稳定版适合做渲染一致的可见表面测距；如果要进一步逼近用户认知中的真实几何表面，需要增加 depth bias、候选模式或二阶段射线校正。
